## Prerequisites

Before you begin, make sure you have the following installed:

| Tool | Version | Notes |
| --- | --- | --- |
| Go | 1.24+ | Required for building from source |
| Node.js + npm | 20+ | Required for building the admin UI |
| Kubernetes | 1.28+ | Locally via [OrbStack](https://orbstack.dev/) or Docker Desktop |
| kubectl | any | For interacting with your cluster |
| helm | v3 | Required for Option A |

<Note>
  ClawHost does **not** need to run inside the Kubernetes cluster. It only needs a kubeconfig that can reach the cluster API.
</Note>

## Deploy ClawHost

Choose the deployment method that best fits your environment:

<Tabs>
  <Tab title="Option A: Helm (recommended)">
    Helm installs ClawHost, PostgreSQL, and all required RBAC resources into your cluster in a single command.

    <Steps>
      <Step title="Build the container image">
        Build the ClawHost image locally and make it available to your cluster:

        ```bash
        docker build -t clawhost:latest .
        ```

        <Note>
          If you're using a remote cluster, push the image to a registry your cluster can pull from, then set `--set server.image.repository=your-registry/clawhost`.
        </Note>
      </Step>

      <Step title="Install with Helm">
        Deploy everything into the `clawhost` namespace:

        ```bash
        helm install clawhost deploy/helm/clawhost \
          -n clawhost --create-namespace \
          --set adminToken="my-admin-token" \
          --set domain.botDomain="clawhost.loc"
        ```

        Key values you can configure at install time:

        | Parameter | Default | Description |
        | --- | --- | --- |
        | `adminToken` | `change-me` | Admin API token — **change this** |
        | `domain.botDomain` | `clawhost.ai` | Domain for bot subdomains and API |
        | `server.image.repository` | `clawhost` | ClawHost container image |
        | `server.image.tag` | `latest` | Image tag |
        | `server.replicas` | `1` | Number of replicas |
        | `postgresql.enabled` | `true` | Deploy bundled PostgreSQL |
        | `postgresql.auth.password` | `postgres` | Database password |
        | `storage.size` | `10Gi` | Shared PVC size for bot data |
        | `openclaw.image` | `1panel/openclaw:latest` | OpenClaw bot image |
        | `openclaw.cpuLimit` | `2000m` | Bot CPU limit |
        | `openclaw.memoryLimit` | `8Gi` | Bot memory limit |
        | `ingress.enabled` | `false` | Enable Ingress |
      </Step>

      <Step title="Verify the deployment">
        Check that the pods are running:

        ```bash
        kubectl -n clawhost get pods
        ```

        Forward the service port and confirm the health endpoint responds:

        ```bash
        kubectl -n clawhost port-forward svc/clawhost 18080:18080
        curl http://localhost:18080/health
        ```

        You should see:

        ```json
        {"status":"ok"}
        ```
      </Step>
    </Steps>

    To use an external database instead of the bundled PostgreSQL:

    ```bash
    helm install clawhost deploy/helm/clawhost \
      -n clawhost --create-namespace \
      --set adminToken="my-admin-token" \
      --set postgresql.enabled=false \
      --set externalDatabase.host="db.example.com" \
      --set externalDatabase.password="secret"
    ```
  </Tab>

  <Tab title="Option B: kubectl apply">
    Apply the raw Kubernetes manifests from `deploy/k8s/` directly.

    <Steps>
      <Step title="Create namespace and RBAC">
        ```bash
        kubectl apply -f deploy/k8s/namespace.yaml
        kubectl apply -f deploy/k8s/rbac.yaml
        kubectl apply -f deploy/k8s/pvc.yaml
        ```
      </Step>

      <Step title="Configure secrets">
        Copy the example secrets file and fill in your tokens and passwords before applying:

        ```bash
        cp deploy/k8s/secrets.yaml.example deploy/k8s/secrets.yaml
        # Edit deploy/k8s/secrets.yaml with your tokens and passwords
        kubectl apply -f deploy/k8s/secrets.yaml
        ```

        <Warning>
          Never commit `deploy/k8s/secrets.yaml` to version control. It contains plaintext credentials.
        </Warning>
      </Step>

      <Step title="Deploy PostgreSQL and ClawHost">
        ```bash
        kubectl apply -f deploy/k8s/postgres.yaml
        kubectl apply -f deploy/k8s/configmap.yaml
        kubectl apply -f deploy/k8s/deployment.yaml
        ```

        The manifests in `deploy/k8s/` cover:

        | File | Description |
        | --- | --- |
        | `namespace.yaml` | Namespace |
        | `rbac.yaml` | ServiceAccount, Role, RoleBinding |
        | `pvc.yaml` | Shared storage for bot data |
        | `postgres.yaml` | PostgreSQL Deployment + Service |
        | `secrets.yaml.example` | Secret template (copy and edit) |
        | `configmap.yaml` | ClawHost `config.toml` |
        | `deployment.yaml` | ClawHost Deployment + Service |
      </Step>

      <Step title="Access ClawHost">
        Forward the service port to reach the API locally:

        ```bash
        kubectl -n clawhost port-forward svc/clawhost 18080:18080
        curl http://localhost:18080/health
        ```
      </Step>
    </Steps>
  </Tab>

  <Tab title="Option C: Local binary">
    Run ClawHost directly on your host machine, connecting to a Kubernetes cluster via kubeconfig. Best for local development.

    <Steps>
      <Step title="Clone and build">
        ```bash
        git clone https://github.com/clawhost/clawhost.git
        cd clawhost
        ```

        Build the admin UI and Go binary in one step:

        ```bash
        make build
        ```

        This runs `npm install && next build` in `web/admin/`, then compiles the Go binary with the admin UI embedded.
      </Step>

      <Step title="Start PostgreSQL">
        ```bash
        docker run -d --name clawhost-pg \
          -e POSTGRES_DB=clawhost \
          -e POSTGRES_USER=postgres \
          -e POSTGRES_PASSWORD=postgres \
          -p 5432:5432 postgres:16
        ```
      </Step>

      <Step title="Prepare the Kubernetes namespace">
        ```bash
        kubectl create namespace clawhost
        kubectl apply -f deploy/k8s/pvc.yaml
        ```
      </Step>

      <Step title="Configure ClawHost">
        Copy the example config and open it in your editor:

        ```bash
        cp config.example.toml config.toml
        ```

        For local development, set at minimum:

        ```toml
        [server]
        port = 18080

        [api]
        admin_token = "my-admin-token"

        [db]
        host = "localhost"
        port = 5432
        user = "postgres"
        password = "postgres"
        database = "clawhost"
        sslmode = "disable"
        timezone = "Asia/Shanghai"

        [kubernetes]
        kubeconfig = ""        # Empty uses ~/.kube/config
        namespace = "clawhost"
        local_dev = true       # Use ClusterIP for direct pod access from host

        [storage]
        pvc_name = "clawhost-shared-data"
        base_path = "/clawhost-data"

        [domain]
        api_domain = "clawhost.ai"
        bot_domain_suffix = "clawhost.ai"
        bot_domain_template = "https://{bot_id}.clawhost.ai"

        [openclaw]
        image = "1panel/openclaw:latest"
        gateway_port = 18789
        cpu_limit = "2000m"
        memory_limit = "8Gi"
        cpu_request = "500m"
        memory_request = "1Gi"
        ```

        <Tip>
          Setting `local_dev = true` under `[kubernetes]` tells ClawHost to use each bot pod's ClusterIP directly instead of DNS, which is necessary when running ClawHost outside the cluster.
        </Tip>
      </Step>

      <Step title="Start the server">
        ```bash
        ./clawhost server
        ```

        You can also pass a custom config path:

        ```bash
        ./clawhost server -c /path/to/config.toml
        ```

        Verify it's running:

        ```bash
        curl http://localhost:18080/health
        ```
      </Step>
    </Steps>
  </Tab>
</Tabs>

## Create your first bot

Once ClawHost is running via any of the options above, follow these steps to provision a bot.

<Steps>
  <Step title="Create an app">
    Each app has its own API token used to authenticate bot operations. Create one with the admin token:

    ```bash
    curl -s -X POST http://localhost:18080/bot/api/v1/admin/apps \
      -H "Authorization: Bearer my-admin-token" \
      -H "Content-Type: application/json" \
      -d '{"name": "my-app"}'
    ```

    Save the `api_token` from the response:

    ```bash
    export API_TOKEN="<api_token from response>"
    ```
  </Step>

  <Step title="Create a bot">
    Create a bot scoped to a user within your app. Pass your AI provider model and API key in the `config` object:

    ```bash
    curl -s -X POST http://localhost:18080/bot/api/v1/bots \
      -H "Authorization: Bearer $API_TOKEN" \
      -H "Content-Type: application/json" \
      -d '{
        "user_id": "user-001",
        "name": "my-bot",
        "slug": "my-bot",
        "config": {
          "model": "claude-sonnet-4-20250514",
          "api_key": "sk-ant-xxx"
        }
      }'
    ```

    Save the `id` from the response:

    ```bash
    export BOT_ID="<id from response>"
    ```
  </Step>

  <Step title="Start the bot">
    Starting a bot provisions a Kubernetes Deployment and Service for it:

    ```bash
    curl -s -X POST http://localhost:18080/bot/api/v1/bots/$BOT_ID/start \
      -H "Authorization: Bearer $API_TOKEN"
    ```
  </Step>

  <Step title="Check bot status">
    Poll the status endpoint until the bot reports as running:

    ```bash
    curl -s http://localhost:18080/bot/api/v1/bots/$BOT_ID/status \
      -H "Authorization: Bearer $API_TOKEN"
    ```
  </Step>

  <Step title="Access the bot via proxy">
    Route requests to your bot through ClawHost's built-in HTTP/WebSocket proxy:

    ```bash
    curl http://localhost:18080/proxy/$BOT_ID/
    ```

    The proxy also supports WebSocket connections on the same path pattern.
  </Step>
</Steps>

### Additional bot operations

<CodeGroup>

```bash Stop a bot
curl -s -X POST http://localhost:18080/bot/api/v1/bots/$BOT_ID/stop \
  -H "Authorization: Bearer $API_TOKEN"
```

```bash Restart a bot
curl -s -X POST http://localhost:18080/bot/api/v1/bots/$BOT_ID/restart \
  -H "Authorization: Bearer $API_TOKEN"
```

```bash Delete a bot
curl -s -X DELETE http://localhost:18080/bot/api/v1/bots/$BOT_ID \
  -H "Authorization: Bearer $API_TOKEN"
```

</CodeGroup>

## Local subdomain routing with Caddy

Bot WebUIs are accessed via subdomains (e.g., `my-bot.clawhost.loc`). For local development, use [Caddy](https://caddyserver.com/) to handle TLS termination and route all subdomains to ClawHost.

<Steps>
  <Step title="Create a Caddyfile">
    Create a `Caddyfile` in the project root:

    ```caddyfile
    # API domain
    clawhost.loc {
        tls internal
        reverse_proxy localhost:18080
    }

    # Bot subdomains
    *.clawhost.loc {
        tls internal
        reverse_proxy localhost:18080
    }
    ```
  </Step>

  <Step title="Run Caddy">
    ```bash
    caddy run
    ```
  </Step>

  <Step title="Configure local DNS">
    Add a static entry to `/etc/hosts` for the base domain:

    ```
    127.0.0.1  clawhost.loc
    ```

    <Warning>
      `/etc/hosts` does not support wildcard entries. For `*.clawhost.loc` to resolve, use [dnsmasq](https://thekelleys.org.uk/dnsmasq/doc.html) or a local DNS resolver. On macOS with OrbStack, you can use `*.orb.local` domains directly without additional configuration.
    </Warning>
  </Step>
</Steps>

## Admin UI

ClawHost includes a built-in web admin panel served at `/admin`. It is a Next.js app statically embedded in the Go binary — no separate deployment required.

Open `http://localhost:18080/admin` in your browser and log in with your `adminToken`.

From the admin UI you can:
- Create, edit, and delete apps; copy or reset API tokens
- List all bots across apps; start, stop, upgrade, and delete bots; view bot domains
