title: Authentication
description: How ClawHost authenticates API requests using admin tokens, app tokens, and per-bot access tokens.

ClawHost uses three distinct token types to control access across its API surface. Each token type is scoped to a different level of the system and validated by a dedicated middleware.

## Token types

<CardGroup cols={3}>
  <Card title="Admin token" icon="shield-check">
    Global operator token. Grants full access to all `/admin/*` routes, including app management and cross-app bot operations.
  </Card>
  <Card title="App token" icon="key">
    Per-app API token. Grants access to all bot routes scoped to that app. Issued on app creation and rotatable on demand.
  </Card>
  <Card title="Bot access token" icon="bot">
    Per-bot token stored in `Bot.AccessToken`. Used for CLI commands and token-based auth to the bot instance itself. Rotatable independently of the app token.
  </Card>
</CardGroup>

## Token format

All tokens are generated using `crypto/rand` and hex-encoded. The `generateSecureToken` function in `model/bot.go` reads `length` random bytes and returns a hex string truncated to that length:

```go
func generateSecureToken(length int) string {
    bytes := make([]byte, length)
    rand.Read(bytes)
    return hex.EncodeToString(bytes)[:length]
}
```

- **App tokens** — 64-character hex string (32 bytes of entropy), generated by `generateAppToken(64)` in `model/app.go`.
- **Bot access tokens** — 32-character hex string (16 bytes of entropy), generated by `generateSecureToken(32)` in `model/bot.go`.

<Note>
All tokens are stored with a `UNIQUE` constraint in the database. App tokens use a `uniqueIndex` on the `api_token` column.
</Note>

## Admin token

The admin token is a static secret set in `config.toml` under `api.admin_token`:

```toml
[api]
admin_token = "your-admin-token"
```

The `AdminAuth` middleware in `middleware/auth.go` validates every request to `/bot/api/v1/admin/*`. It reads the configured token via `viper.GetString("api.admin_token")` and compares it to the `Authorization` header:

```go
// AdminAuth returns a middleware that validates admin token from config
func AdminAuth() echo.MiddlewareFunc {
    return func(next echo.HandlerFunc) echo.HandlerFunc {
        return func(c echo.Context) error {
            adminToken := viper.GetString("api.admin_token")
            if adminToken == "" {
                return util.Unauthorized(c, "admin access is disabled")
            }
            // ...
            if token != adminToken {
                return util.Unauthorized(c, "invalid admin token")
            }
            return next(c)
        }
    }
}
```

<Warning>
If `api.admin_token` is empty, the server returns `401 Unauthorized` for all admin routes — admin access is effectively disabled. Always set a strong token in production.
</Warning>

### Admin token usage

```bash
# Create an app
curl -X POST https://clawhost.example.com/bot/api/v1/admin/apps \
  -H "Authorization: Bearer your-admin-token" \
  -H "Content-Type: application/json" \
  -d '{"name": "my-app"}'

# List all apps
curl https://clawhost.example.com/bot/api/v1/admin/apps \
  -H "Authorization: Bearer your-admin-token"

# Verify the admin token
curl https://clawhost.example.com/bot/api/v1/admin/verify \
  -H "Authorization: Bearer your-admin-token"
```

## App token

An app token is automatically generated when an app is created via `POST /bot/api/v1/admin/apps`. It is returned in the `api_token` field of the response and is the credential used for all bot-level API calls.

The `BearerAuth` middleware in `middleware/auth.go` validates app tokens by looking them up in the `apps` table:

```go
// First, try to validate against apps table
app, err := model.GetAppByAPIToken(token)
if err == nil && app != nil {
    // Token found in apps table, store app in context
    c.Set(ContextKeyApp, app)
    return next(c)
}
```

Only apps with `status = 'active'` are matched. A disabled app's token is rejected.

### App token usage

```bash
# Create an app and capture its token
APP=$(curl -s -X POST https://clawhost.example.com/bot/api/v1/admin/apps \
  -H "Authorization: Bearer your-admin-token" \
  -H "Content-Type: application/json" \
  -d '{"name": "my-app"}')

export API_TOKEN=$(echo $APP | jq -r '.data.api_token')

# Create a bot using the app token
curl -X POST https://clawhost.example.com/bot/api/v1/bots \
  -H "Authorization: Bearer $API_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"user_id": "user-001", "name": "my-bot", "slug": "my-bot"}'

# List bots scoped to this app
curl https://clawhost.example.com/bot/api/v1/bots \
  -H "Authorization: Bearer $API_TOKEN"
```

## Bot access token

Each bot has its own `access_token` field (type `varchar(64)`). It is generated automatically on bot creation and stored in the `Bot` model:

```go
type Bot struct {
    // ...
    AccessToken string `json:"access_token" gorm:"type:varchar(64)"` // Used for CLI commands and token auth
    // ...
}
```

The bot access token is used by OpenClaw CLI commands and for token-based authentication to the bot's gateway (via `gateway.auth.token` in the OpenClaw config). When the token is reset, the gateway config inside the running pod is updated automatically.

## Resetting tokens

Both app tokens and bot access tokens can be rotated without recreating the resource.

### Reset an app token

Requires the admin token. Issues a new 64-character token and invalidates the old one immediately.

```bash
curl -X POST https://clawhost.example.com/bot/api/v1/admin/apps/{id}/reset-token \
  -H "Authorization: Bearer your-admin-token"
```

Response:

```json
{
  "code": 0,
  "data": {
    "api_token": "<new-64-char-token>"
  }
}
```

### Reset a bot access token

Requires the app token for the app that owns the bot. If the bot is currently running, the `ResetBotToken` handler in `handler/api/v1/bot_reset_token.go` also pushes the new token into the live Kubernetes Deployment so the pod picks it up without a manual restart.

```bash
curl -X POST https://clawhost.example.com/bot/api/v1/bots/{id}/reset-token \
  -H "Authorization: Bearer $API_TOKEN"
```

Response:

```json
{
  "code": 0,
  "data": {
    "id": "<bot-id>",
    "access_token": "<new-32-char-token>",
    "access_url": "https://<slug>.clawhost.example.com?token=<new-token>",
    "message": "access token has been reset"
  }
}
```

<Note>
Resetting the bot access token also updates `gateway.auth.token` inside the bot's OpenClaw config in the database and, for running bots, patches the Kubernetes Deployment spec so the change takes effect immediately.
</Note>

## Middleware chain

The following middleware functions in `middleware/auth.go` are applied to each route group:

| Middleware | Route group | What it validates |
|---|---|---|
| `BearerAuth()` | `/bot/api/v1/*` | App token against the `apps` table |
| `BotOwnerAuth()` | `/bot/api/v1/bots/:id/*` | App in context owns the requested bot (`bot.app_id == app.id`) |
| `AdminAuth()` | `/bot/api/v1/admin/*` | Token matches `api.admin_token` from config |

`BotOwnerAuth` runs after `BearerAuth` and reads the app stored in context. If the bot's `app_id` does not match the authenticated app's `id`, the request is rejected with `403 Forbidden`.

## Authorization header format

All authenticated endpoints expect a standard HTTP Bearer token header:

```
Authorization: Bearer <token>
```

The middleware rejects requests that:
- Omit the `Authorization` header entirely (returns `401`)
- Use a scheme other than `Bearer` (case-insensitive) (returns `401`)
- Supply an unrecognised token (returns `401`)
- Supply a valid app token for a bot owned by a different app (returns `403`)

## Security best practices

<Steps>
  <Step title="Use HTTPS in production">
    Tokens are transmitted in HTTP headers as plaintext. Always terminate TLS before traffic reaches ClawHost in production. The example Caddyfile in the README provides a reference configuration with automatic internal TLS.
  </Step>
  <Step title="Set a strong admin_token">
    The `api.admin_token` is a static secret. Use a long, randomly generated value (at minimum 32 characters). Never reuse this value across environments.

    ```bash
    # Generate a strong admin token
    openssl rand -hex 32
    ```
  </Step>
  <Step title="Rotate tokens regularly">
    Use `POST /admin/apps/:id/reset-token` and `POST /bots/:id/reset-token` to rotate credentials after suspected exposure or on a scheduled basis. Old tokens are invalidated immediately upon reset.
  </Step>
  <Step title="Disable apps instead of deleting them">
    Set `status` to `disabled` on an `App` to block all requests using that app's token without deleting the app's bots. Tokens belonging to disabled apps are rejected by `GetAppByAPIToken`, which filters on `status = 'active'`.
  </Step>
  <Step title="Store tokens in secrets, not config files">
    When deploying to Kubernetes, store `api.admin_token` and database credentials in a Kubernetes Secret rather than baking them into a ConfigMap or image. The provided `deploy/k8s/secrets.yaml.example` shows the expected structure.
  </Step>
</Steps>
