## Base URL

All API endpoints are served on port `18080`:

```
http://<host>:18080
```

All bot and app routes are prefixed with `/bot/api/v1`.

## Authentication

Most routes require an `Authorization` header with a Bearer token:

```bash
Authorization: Bearer <token>
```

ClawHost uses two types of tokens:

| Token type | Used for | Configured via |
| --- | --- | --- |
| **Admin token** | `/bot/api/v1/admin/*` routes | `api.admin_token` in `config.toml` |
| **App token** | `/bot/api/v1/bots/*` routes | Generated per app, returned on app creation |

The **admin token** is a static secret you configure when deploying ClawHost. The **app token** (`api_token`) is generated automatically when you create an app and scoped to that app's bots.

<Note>
  The `GET /health` endpoint does not require authentication.
</Note>

## Health check

```
GET /health
```

No authentication required. Returns `200 OK` with a JSON status body when the server is running.

```json
{ "status": "ok" }
```

## Response format

All responses are JSON. Successful responses follow this envelope:

```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

Error responses include an HTTP error status code and a JSON body with details:

```json
{
  "code": 400,
  "message": "name is required"
}
```

## Error codes

| HTTP status | Meaning |
| --- | --- |
| `400` | Bad request — invalid or missing parameters |
| `401` | Unauthorized — missing or invalid token |
| `403` | Forbidden — token valid but not authorized for this resource |
| `404` | Not found — resource does not exist |
| `500` | Internal server error |

## Rate limiting

The ClawHost API does not enforce rate limiting at the application level. Apply rate limiting at your load balancer or ingress layer if needed.

### Proxy

No authentication required at the proxy layer. The target bot enforces its own access control.

| Method | Path | Description |
| --- | --- | --- |
| `ANY` | `/proxy/:bot_id/*` | HTTP proxy to bot instance |
| `WS` | `/proxy/:bot_id/*` | WebSocket proxy to bot instance |

Subdomain routing is also supported: requests to `{bot-id}.{bot_domain}/*` are automatically routed to the corresponding bot without a path prefix.
